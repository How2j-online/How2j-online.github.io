三、进程管理

1. 概念：

  • 进程是正在执行的程序的实例，是一个动态的概念，是程序的一次执行过程，是系统进行资源分配和调度的一个独立单位。
  • 进程有自己的地址空间， 占用一定的系统资源。
  • 一个CPU在同一时刻只能执行一个进程，多个进程并发执行。
  • 进程由它的进程ID(PID)和它的父进程ID(PPID)来唯一标识。

declare 命令

shell中所有变量默认都是字符串，使用declare命令可以显示的声明变量的类型，包括整型、字符串、数组等。

3. 环境变量

$* 传递给脚本或函数的所有参数 for i in $*

$HOME 用户主文件夹 cd $HOME

4. 运算符和引用

类型 符号 作用 用法
算术运算符 + - * / % = 算术运算 let a=1+2
逻辑运算符 && \|\| ! 逻辑运算 if [ $a -gt 1 ] && [ $b -gt 1 ]
比较运算符 == != < > 比较运算 if [ $a -gt 1 ]
引号 双引号 "" 部分引用，仅$`\保留作用 foo="${a}213"
单引号'' 完全引用，原样输出 foo='foo$a'
反引号 `` 执行命令 foo=`ls -a`
圆括号 (()) 整数运算 foo=$((1+2))
() 执行命令 foo=$(ls -a)
命令连接 || cmd1执行完且返回码非0，则继续执行cmd2 cmd1 || cmd2
&& cmd1执行完且返回码为0，则继续执行cmd2 cmd1 && cmd2
; cmd1、cmd2串行执行 cmd1;cmd2
后台执行 & 可与nohup一起使用 cmd &

7. 判断命令

语法： test condition 或者 [ condition ] 或者 [[ condition ]]

注意：

  • 中括号两边必须有空格
  • [ 和 test是命令，只能使用自己支持的标志位，<、>、=只能用来比较字符串，-lt、-gt、-eq只能用来比较数字
  • 中括号内的变量必须要用双引号引起来，否则如果变量为空，会报错
  • [[ 更丰富，在整形比较中支持<、>、=，在字符串比较中支持==、!=、正则，并且不需要引号引起来

3. 参数展开

  1. 间接参数扩展${!parameter}，其中引用的参数并不是parameter而是parameter的实际的值
  2. 参数长度 ${#parameter}
  3. 空参数处理
    • ${parameter:=-word} 为空替换
    • ${parameter:=word} 为空赋值，如果parameter为空，则赋值为word
    • ${parameter:?word} 为空报错，如果parameter为空，则报错，错误信息为word
    • ${parameter:+word} 为空忽略，如果parameter为空，则忽略word
  4. 参数切片
    • ${parameter:offset} 从offset开始截取
    • ${parameter:offset:length} 从offset开始截取，截取长度为length
  5. 参数部分删除
    • ${parameter#word} 从parameter的开头开始，删除最短匹配的word
    • ${parameter##word} 从parameter的开头开始，删除最长匹配的word
    • ${parameter%word} 从parameter的结尾开始，删除最短匹配的word
    • ${parameter%%word} 从parameter的结尾开始，删除最长匹配的word

6. 文件名展开

文件名展开是指将通配符展开为文件名，通配符包括：* ? [ ]

当有单词没有被引号包围时且出现了 ‘*’, “?”, and “[“ 字符，则Shell会去按照正则匹配的方式查找文件名，如果找到了，则将其替换为文件名

编程范式

概念：编程范式是一种编程风格或方法论，用于指导程序员在解决问题时如何组织和表达代码。它是一种抽象的概念，描述了计算机编程中的一系列原则和规则，以及程序设计的思维方式。不同的编程范式强调不同的概念和技术，以适应不同类型的问题和场景。根据程序语言的特性，如是否允许副作用，操作的执行顺序，代码组织，状态管理，语法和词法等，可以将编程范式分为不同的类型。

一、命令式

1. 面向过程

2. 面向对象

概念

  • 封装: 关联数据与算法
  • 继承: 无需重写的情况下进行功能扩展
  • 多态: 不同的结构可以进行接口共享，进而达到函数重用
  • 依赖注入: 去除代码耦合

五大基本原则

  • 单一职责原则(SRP-Single Responsibility Principle): 一个类只负责一个功能领域中的相应职责
  • 开闭原则(OCP-Open Close Principle): 软件实体应对扩展开放，而对修改关闭
  • 里氏替换原则(LSP-the LisKov Substitution Principle LSP): 所有引用基类的地方必须能透明地使用其子类的对象
  • 依赖倒置原则(DIP-the Dependency Inversion Principle DIP): 抽象不应该依赖于细节，细节应该依赖于抽象
  • 接口隔离原则(ISP-the Interface Segregation Principle ISP): 使用多个专门的接口，而不使用单一的总接口

客户端容器

一、浏览器架构

单进程架构: 所有模块运行在同一个进程里，包含网路、插件、JavaScript运行环境

多进程架构: 主进程、网络进程、渲染进程、CPU进程、插件进程

面向服务架构: 将原来的UI、数据库、文件、设备、网路等，作为一个独立的基础服务

架构对比：

构架类型 扩展性 安全性 稳定性 流程度
单进程架构 低，所有模块运行在同一进程里，访问同一块内存区域，数据没有隔离，新增模块可能会影响原有功能 低，三方插件可直接访问操作系统任意资源 低，三方插件漏洞或者某个tab页面JavaScript脚本问题可能导致浏览器崩溃 卡顿，所有页面运行在同一进程中，开启多个页面明显卡顿
多进程架构 中，各进程分配独立的内存区域，有些进程功能较大，耦合度高 高，运行在独立沙箱中，不能访问系统敏感资源 高，进程相互隔离，当一个页面或者插件崩溃时，不会影响其他进程 流程，每个页面运行在独立的渲染进程中，充分利用系统资源
面向服务架构 高，服务模块划分更细，更内聚，耦合性低，易于扩展 高，运行在独立沙箱中，不能访问系统敏感资源 高，进程相互隔离，当一个页面或者插件崩溃时，不会影响其他进程 流程，每个页面运行在独立的渲染进程中，充分利用系统资源

二、多进程的分工

进程名称 描述
浏览器(主进程) 主要赋值页面展示逻辑，用户交互，子进程管理，包括地址栏，书签，前进，后退，收藏夹等功能
GPU进程 负责UI绘制，包含整个浏览器全部UI
网络进程 用于处理网络请求，如HTTP请求，文件下载等，主要是将请求后的数据传递给渲染进程
标签页(渲染进程) 控制tab内的所有内容，将html，css和JavaScript转换为用户可交互的网页
插件进程 控制网站运行的插件，比如flash，modheader等
其他进程 适用程序Storage/Network/Audio Service等

三、渲染进程

1. 渲染进程的组成

线程 功能
GUI渲染线程 负责渲染浏览器界面，解析HTML，CSS，构建DOM树和RenderObject树，布局和绘制等，和Js引擎线程互斥，GUI更新会在Js引擎空闲时立即执行
Js引擎 负责解析Js脚本，运行Js程序，每个渲染进程下面只有一个Js引擎线程，与GUI渲染线程互斥，如果Js任务执行事件过长，会导致页面卡顿
定时器触发 定时器所在线程， stTimeout，setInterval计时完毕后，将回调添加到事件队列，等待Js引擎执行
网络线程 在XHR，Fetch等发起请求后新开一个网络线程请求，如果设置了回调函数，在状态变更时，将回调放入事件队列，等待Js引擎执行
事件触发 右宿主环境提供，用于控制时间循环，不断从事件队列里取出任务执行

2. 渲染进程的工作流程

  1. 网络线程负责加载网页资源
  2. Js引擎解析Js脚本并执行
  3. Js解析引擎空闲时，渲染线程立即工作
  4. 用户交互，定时器操作等产生回调函数放入任务队列中
  5. 事件线程进行事件循环，将队列里的任务取出交给Js引擎执行
